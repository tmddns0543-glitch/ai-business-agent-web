# AI Business Agent Web Information Architecture

## 1. 문서 목적

이 문서는 AI Business Agent Web의 전체 정보 구조와 화면 계층을 정의한다.

Frontend 개발은 이 문서를 기준으로 진행한다.

현재 MVP는 외식업 점주의 하루 마감을 돕는 모바일 중심 웹서비스이며,
매출 입력, 예상 정산 계산, 비용 확인, 배달대행사 확인을 핵심 흐름으로 한다.

모든 화면은 모바일 사용을 우선으로 설계한다.

---

# 2. 비회원 화면

현재 MVP 이후 구현 예정 영역이다.

- 서비스 소개
- 로그인
- 회원가입
- 비밀번호 재설정
- 베타 이용 안내

현재 LocalStorage 기반 프로토타입에서는 비회원·회원 구분을 적용하지 않는다.

- 전체 진행률 확인
- 마감 완료 처리

마감 단계:

1. 매출 확인
2. 비용 확인
3. 배달대행사 확인

진행률 기준:

```text
매출만 완료 → 33%
매출 + 비용 완료 → 67%
매출 + 비용 + 배달대행사 완료 → 100%
```

정산 계산은 새로운 마감 단계가 아니다.

정산 결과는 매출 단계의 결과로 표시한다.

매출 카드 표시 정보:

- 총매출
- 예상 공제액
- 예상 정산금액
- 매출 확인 완료 상태

예상 공제액과 예상 정산금액은 플랫폼별 매출과 수수료 설정을 기준으로 계산한다.

## 4.3 주문 수 처리

주문 수는 매출액에 포함하지 않는다.

- 배민원
- 쿠팡이츠 배달
- 요기배달

예시:

```text

건당 배달료 3,400원

총매출 = 100,000원
배달료 = 34,000원
```

주문 수 10건을 총매출에 더하지 않는다.

가게배달 현금
→ 결제수수료와 카드수수료 모두 미적용

배민과 요기요의 가게배달 카드결제에도 중개수수료는 적용한다.

# 12. 수수료 설정 UX

각 플랫폼 설정 화면은 다음 기능을 제공한다.

- 현재 설정 불러오기
- 입력값 수정
- 전체 입력 검증
- 저장
- 새로고침 후 복원
- 해당 플랫폼만 기본값 복원
- 정산 예상금액 확인

수수료율 입력:

```text
0 이상
일반 수수료율 최대 100
카드수수료율 최대 1.5
소수 허용
```

건당 배달료 입력:

```text
0 이상의 정수
```

빈 값은 저장 시 0으로 처리한다.

잘못된 입력이 하나라도 있으면 해당 플랫폼의 전체 저장을 막는다.

플랫폼 기본값 복원은 다른 플랫폼 설정에 영향을 주지 않는다.

# 17. 정보 구조 원칙

모든 기능은 다음 순서를 따른다.

```text
입력
↓
저장
↓
계산
↓
결과
↓
확인
```

사용자는 내부 수수료 계산 구조를 이해할 필요가 없다.

사용자에게 필요한 것은 다음 세 가지다.

```text
얼마를 팔았는가
얼마가 빠지는가
얼마가 들어올 것인가
```

따라서 핵심 화면에서는 다음 정보를 우선한다.

```text
총매출
예상 공제액
예상 정산금액
```

서비스는 내부 구조보다 사용자의 다음 행동이 먼저 보이도록 설계한다.
