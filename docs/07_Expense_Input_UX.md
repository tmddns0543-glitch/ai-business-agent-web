# AI Business Agent Web Expense Input UX

## 1. 문서 목적

이 문서는 웹서비스의 비용입력 구조, 입력 흐름, 저장 방식, 수정 방식과 일별 확인 방식을 정의한다.

비용은 매출과 달리 하루 한 번의 합계 데이터가 아니라, 발생한 거래를 한 건씩 기록하는 방식으로 저장한다.

핵심 원칙:

> 매출은 하루를 기록하는 데이터이고, 비용은 거래를 기록하는 데이터이다.

---

# 2. 핵심 UX 목표

비용입력은 다음 조건을 만족해야 한다.

- 비용이 발생할 때마다 바로 입력할 수 있어야 한다.
- 하루가 끝난 뒤 여러 건을 연속해서 입력할 수도 있어야 한다.
- 한 건 입력에 30초 이상 걸리지 않아야 한다.
- 사용자가 자주 쓰는 비용항목을 빠르게 선택할 수 있어야 한다.
- 거래처, 금액, 메모를 간단히 남길 수 있어야 한다.
- 오늘 입력한 비용을 한눈에 확인하고 수정할 수 있어야 한다.
- 입력한 거래는 월별 경영성과에 자동 반영되어야 한다.
- 향후 영수증 OCR, 카드 연동, 계좌 연동으로 확장 가능해야 한다.

---

# 3. 비용 데이터 단위

비용은 한 건의 거래 단위로 저장한다.

예시:

- 2026-07-12 / 원재료비 / 짜르푸드 / 187,000원
- 2026-07-12 / 배달대행 충전 / 바로고 / 200,000원
- 2026-07-12 / 광고비 / 배달의민족 / 30,000원
- 2026-07-12 / 소모품비 / 다이소 / 18,500원

한 날짜에 여러 건의 비용이 저장될 수 있다.

---

# 4. 비용입력 진입 방식

사용자는 다음 경로에서 비용입력으로 진입할 수 있다.

- 하단 메뉴의 비용입력
- 홈의 오늘 비용 추가
- 매출 정산 완료 화면의 비용 입력하기
- 오늘 비용 목록의 비용 추가

진입 시 가장 먼저 빠른 비용입력 화면을 보여준다.

---

# 7. 금액 및 거래방향 입력 규칙

## 7.1 기본 원칙

비용 금액은 사용자가 항상 0보다 큰 양수로 입력한다.

사용자가 직접 음수 부호를 입력하는 방식은 사용하지 않는다.

거래의 증가·감소 방향은 금액이 아니라 거래유형으로 구분한다.

예시:

- 일반 지출 30,200원
- 환급 4,000원
- 현금주문 캐시입금 50,000원
- 잔액 조정 10,000원

시스템은 거래유형에 따라 비용과 배달대행사 예수금에 미치는 영향을 자동으로 계산한다.

---

## 7.2 일반 비용 거래

일반 비용 거래는 다음과 같이 처리한다.

### 일반 지출

예시:

배달대행비 30,200원

처리 결과:

- 비용 +30,200원
- 배달대행사 예수금 -30,200원

### 환급 또는 매입 취소

예시:

배달대행비 환급 4,000원

처리 결과:

- 비용 -4,000원
- 배달대행사 예수금 +4,000원

사용자는 금액을 4,000원으로 입력하고,
프로그램이 환급 거래유형을 기준으로 비용 차감과 잔액 증가를 자동 처리한다.

---

## 7.3 현금주문 캐시입금

플랫폼 주문이 현금결제로 발생했으나,
배달기사가 매장에 현금을 전달하지 못하고
해당 물건값을 배달대행사 캐시 또는 예수금으로 지급하는 경우가 있다.

이 거래는 비용 환급이 아니라,
현금으로 받을 매출대금을 배달대행사 캐시로 받은 정산 거래다.

예시:

- 배달대행비 사용 30,200원
- 현금주문 캐시입금 50,000원

배달대행사 예수금 계산:

전일 예수금
- 금일 배달대행비
+ 현금주문 캐시입금

예시 계산:

100,000원
- 30,200원
+ 50,000원
= 119,800원

전일 대비 예수금 순증가:

+19,800원

현금주문 캐시입금의 처리 결과:

- 배달대행사 예수금 +50,000원
- 비용 영향 없음
- 매출 추가 반영 없음

현금 주문 매출은 매출입력에서 이미 반영되므로,
캐시입금 시 다시 매출로 계산하지 않는다.

---

## 7.4 배달대행사 거래유형

배달대행사 관련 거래는 다음과 같이 구분한다.

### 캐시 충전

매장이 배달대행사 캐시를 충전한 거래다.

처리 결과:

- 배달대행사 예수금 증가
- 비용 영향 없음

### 배달비 사용

배달 수행으로 캐시가 차감된 거래다.

처리 결과:

- 배달대행사 예수금 감소
- 배달대행 비용 증가

### 현금주문 캐시입금

배달기사가 전달해야 할 현금 주문대금을
배달대행사 캐시로 대신 지급한 거래다.

처리 결과:

- 배달대행사 예수금 증가
- 비용 영향 없음
- 매출 영향 없음

### 환급 또는 취소

잘못 차감된 배달비나 취소된 거래금액이 반환된 경우다.

처리 결과:

- 배달대행사 예수금 증가
- 배달대행 비용 감소

### 잔액 조정

실제 예수금과 프로그램 잔액의 차이를 맞추는 거래다.

처리 결과:

- 배달대행사 예수금 증가 또는 감소
- 비용 영향 여부는 조정 사유에 따라 결정

---

## 7.5 내부 계산 방향

사용자는 모든 금액을 양수로 입력한다.

프로그램은 거래유형에 따라 내부 방향을 적용한다.

예시:

- 일반 지출 → 비용 증가
- 환급 → 비용 감소
- 캐시 충전 → 예수금 증가
- 배달비 사용 → 예수금 감소 및 비용 증가
- 현금주문 캐시입금 → 예수금 증가
- 배달비 환급 → 예수금 증가 및 비용 감소
- 잔액 조정 → 선택한 방향에 따라 예수금 증가 또는 감소

배달대행사 예수금 계산식:

현재 예수금
= 전일 예수금
+ 캐시 충전
+ 현금주문 캐시입금
+ 환급
± 잔액 조정
- 배달비 사용

---

## 7.6 입력 화면 규칙

비용 또는 배달대행사 거래 입력 시 다음 순서로 입력한다.

1. 거래유형 선택
2. 금액 입력
3. 배달대행사 선택
4. 날짜 선택
5. 필요 시 관련 매출 날짜 선택
6. 메모 입력

사용자는 금액 입력창에 음수를 직접 입력할 수 없다.

잔액 조정 거래의 경우에만 다음 중 하나를 선택한다.

- 잔액 증가
- 잔액 감소

---

## 7.7 금액 입력 검증

- 금액은 0보다 큰 정수만 허용한다.
- 음수는 직접 입력할 수 없다.
- 숫자 입력 즉시 천 단위 구분기호를 표시한다.
- 모바일에서는 숫자 키패드를 사용한다.
- 쉼표가 포함된 숫자를 붙여넣어도 정상 처리한다.
- 0원 거래는 저장하지 않는다.
- 거래유형과 금액은 필수 입력값이다.

---

# 8. 거래처 입력

거래처 입력은 선택사항이다.

다음 기능을 제공한다.

- 최근 거래처 추천
- 비용항목별 자주 쓰는 거래처 추천
- 직접 입력
- 신규 거래처 저장

예시:

원재료비 선택 시 추천:

- 짜르푸드
- 롯데칠성
- 농협마트

같은 거래처명을 반복 입력하지 않도록 자동완성을 제공한다.

---

# 9. 메모 입력

메모는 선택사항이다.

메모 예시:

- 닭고기 3박스
- 7월 임대료
- 배민 광고상품
- 배달대행 충전
- 주방 수도꼭지 교체

향후 영수증 OCR 결과와 연결할 수 있다.

---

# 10. 저장 방식

## 10.1 저장하고 하나 더 입력

비용을 저장한 뒤 다음 입력 화면을 바로 비운다.

유지할 수 있는 값:

- 날짜
- 최근 비용항목
- 최근 거래처

연속 입력에 적합하다.

## 10.2 저장하고 끝내기

비용을 저장한 뒤 오늘 비용 목록으로 이동한다.

---

# 11. 오늘 비용 목록

## 11.1 화면 목적

오늘 입력한 비용을 한눈에 확인하고 수정한다.

## 11.2 상단 요약

표시 내용:

- 오늘 비용 총액
- 입력 건수
- 가장 큰 비용
- 마지막 입력 시각

예시:

오늘 비용  
455,500원

총 4건

각 거래를 누르면 상세화면으로 이동한다.

---

# 12. 일괄 입력 UX

하루 마지막이나 다음날 아침 여러 건을 입력하는 사용자를 위해 연속 입력을 지원한다.

흐름:

비용입력
→ 첫 거래 저장
→ 저장하고 하나 더 입력
→ 두 번째 거래 저장
→ 반복
→ 오늘 비용 목록 확인
→ 완료

별도의 복잡한 일괄 입력표를 초기 MVP에서 만들지 않는다.

건별 입력을 반복하는 방식으로 두 상황을 모두 만족시킨다.

향후 베타 피드백에서 필요성이 확인되면 표 형태의 일괄 입력을 추가한다.

---

# 13. 기존 거래 수정

비용 목록에서 거래를 선택하면 수정 화면을 연다.

수정 가능 항목:

- 날짜
- 비용항목
- 금액
- 거래처
- 메모

저장 전 변경 내용을 보여줄 수 있다.

예시:

변경 전  
원재료비 187,000원

변경 후  
원재료비 178,000원

버튼:

- 수정 저장
- 변경 취소

---

# 14. 거래 삭제

삭제는 상세화면 하단의 위험영역에 배치한다.

안내:

이 비용을 삭제하면 월별 경영성과에서도 제외됩니다.

확인 방식:

- 삭제 확인 체크
- 최종 삭제 버튼

삭제된 데이터는 일정 기간 복구 가능한 상태로 보관하는 것을 원칙으로 한다.

---

# 15. 비용 없음 확인

사용자가 하루 마감 시 비용이 없었던 날도 상태를 확정할 수 있어야 한다.

홈의 오늘 비용 카드에서 다음 기능을 제공한다.

버튼:

오늘 발생한 비용 없음

이 상태는 단순 미입력과 구분한다.

상태 예시:

- 확인 전
- 비용 입력 4건
- 발생 비용 없음

---

# 16. 홈 화면 반영

홈의 오늘 비용 카드는 다음 정보를 표시한다.

비용 입력 전:

오늘 비용  
확인 전

버튼: 오늘 비용 확인하기

비용이 있는 경우:

오늘 비용

버튼: 비용 추가 또는 확인

비용이 없는 경우:

오늘 비용  
발생 비용 없음

# 17. 경영성과 반영

저장된 비용 거래는 다음 기준으로 자동 집계한다.

- 날짜별
- 월별
- 비용 대분류별
- 비용항목별
- 거래처별
- 고정비·변동비별
- 원가성 비용 여부별

경영성과에서는 한 건씩 입력한 거래를 합산해 월별 비용을 계산한다.

---

# 18. 고정비 처리

임대료, 급여, 보험료처럼 반복되는 비용도 초기 MVP에서는 거래 단위로 저장한다.

향후 다음 기능으로 확장한다.

- 반복 비용 등록
- 매월 자동 생성
- 결제 여부 확인
- 금액 변경 알림

초기 웹 MVP에서는 자동 반복 생성 기능을 제외한다.

---

# 19. 배달대행 충전

배달대행 충전은 일반 비용 거래이면서 배달대행사 현황에도 반영한다.

입력항목:

- 날짜
- 배달대행사
- 충전금액
- 메모

배달대행 충전으로 저장하면:

- 비용에 반영
- 배달대행사 예수금 증가
- 배달대행사 현황에 기록

중복 반영되지 않도록 하나의 거래를 두 화면에서 공유한다.

# 20. 오류 처리

## 저장 실패

입력값은 유지한다.

안내:

비용을 저장하지 못했습니다.

입력한 내용은 그대로 유지되고 있습니다.

버튼:

- 다시 저장
- 임시저장 후 나가기

## 네트워크 오류

기기에 임시저장한다.

안내:

인터넷 연결이 불안정합니다.

입력한 내용은 기기에 임시저장되었습니다.

연결이 복구되면 다시 저장해주세요.

# 21. 향후 확장

비용 거래 구조는 다음 기능으로 확장 가능해야 한다.

- 세금계산서 연동

# 22. 성공 기준

비용입력 UX는 다음 기준을 만족해야 한다.

- 한 건 입력 30초 이내
- 하루 5건 입력 3분 이내
- 연속 입력 가능
- 발생 즉시 입력과 하루 마감 입력 모두 가능
- 거래 수정과 삭제가 쉬움
- 입력 즉시 오늘 비용 합계 반영
- 월별 경영성과에 정확히 집계
