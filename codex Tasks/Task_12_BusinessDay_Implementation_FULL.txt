다음 구조를 만든다.

 ↓

# 1. 신규 타입 파일

다음 파일을 생성한다.

frontend/src/types/business-day.ts

다음 역할의 타입을 정의한다.

export type BusinessDate = string;

export interface BusinessDayContext {
  version: 1;
  selectedBusinessDate: BusinessDate;
}

BusinessDate는 런타임에서 YYYY-MM-DD 형식을 검증할 수 있어야 한다.

다음 함수 또는 동일 역할의 유틸리티를 함께 정의할 수 있다.

isValidBusinessDate(value: unknown): value is BusinessDate

검증 조건:

- 문자열
- YYYY-MM-DD 형식
- 실제로 존재하는 날짜

예:

- 2026-07-16 → 유효
- 2026-02-30 → 무효
- 2026-7-1 → 무효
- hello → 무효

UTC 변환 때문에 날짜가 하루 밀리지 않도록 주의한다.

## 구현 함수

### getTodayBusinessDate()

브라우저 또는 로컬 시간 기준 오늘 날짜를 YYYY-MM-DD로 반환한다.

toISOString()을 그대로 사용해 UTC 날짜를 가져오지 않는다.

사용자 로컬 날짜 기준이어야 한다.

### getSelectedBusinessDate()

역할:

LocalStorage 조회
↓
정상 구조·버전·날짜 검증
↓
정상이면 selectedBusinessDate 반환
↓
없거나 잘못되면 오늘 날짜 반환

서버 환경에서는 LocalStorage에 접근하지 않고 오늘 날짜를 반환한다.

### setSelectedBusinessDate(date)

역할:

날짜 검증
↓
정상일 때만 LocalStorage 저장
↓
저장 성공 여부 반환

잘못된 날짜는 저장하지 않는다.

### getBusinessDayContext()

필요하다면 완성된 BusinessDayContext를 반환하는 함수도 제공한다.

## 오류 처리

다음 경우 예외 없이 오늘 날짜를 반환한다.

- key 없음
- JSON 파싱 실패
- null
- 배열
- 문자열
- version 없음
- version 불일치
- selectedBusinessDate 없음
- 잘못된 날짜
- LocalStorage 접근 오류
- 서버 환경

잘못된 저장값을 임의로 삭제하지 않는다.

## 현재 하드코딩 날짜 제거

현재 화면에 하드코딩된 2026년 7월 12일 영업일 또는 유사 문구를 제거한다.

## 선택 영업일 state

마운트 후 getSelectedBusinessDate()로 현재 영업일을 읽어 state에 저장한다.

Hydration 오류가 없도록 한다.

## 날짜 선택 UI

마감 화면 상단에 영업일 선택 영역을 만든다.

권장 형태:

영업일
[ 2026-07-16 ]

HTML input type="date"를 사용해도 된다.

모바일에서 사용하기 쉬워야 한다.

선택한 날짜가 바뀌면 setSelectedBusinessDate()로 저장하고 현재 state를 갱신한다.

이번 Task에서는 날짜 변경 시 매출이나 완료 상태를 다시 불러오지 않는다.

다만 다음 안내를 작은 문구로 표시한다.

선택한 영업일 기준으로 매출·비용·마감 데이터를 확인합니다.

## 날짜 표시 형식

사용자 화면에서는 다음처럼 표시한다.

2026년 7월 16일 영업일

입력값은 내부적으로 2026-07-16을 사용한다.

날짜 포맷 함수는 파일 내부 또는 공통 유틸로 둘 수 있다.

# 6. 날짜 변경 동작

이번 Task의 범위에서는 날짜 변경 시 다음만 보장한다.

/closing에서 날짜 선택
↓
LocalStorage 저장
↓
/closing/sales에서 같은 날짜 표시
↓
플랫폼 입력 화면에서 같은 날짜 표시
↓
새로고침 후 유지

아직 실제 매출 저장값은 날짜별로 바뀌지 않는다.

따라서 작업 결과 보고에는 다음 한계를 명확히 적는다.

- 현재 영업일 선택은 화면 공유 기반만 구현됨
- 기존 sales-* 데이터는 아직 날짜별로 분리되지 않음

# 7. 서버·클라이언트 안전성

다음을 지킨다.

- 서버 환경에서 window 접근 금지
- 서버 환경에서 localStorage 접근 금지
- Hydration mismatch 방지
- 잘못된 날짜에서 화면 오류 금지
- NaN 날짜 표시 금지
- Invalid Date 표시 금지

필요한 페이지는 "use client"를 사용한다.

이미 클라이언트 컴포넌트라면 중복 추가하지 않는다.

- 예상 정산금액

- closing-sales-confirmed

아직 날짜별 마감 상태가 없기 때문이다.

# 9. 필수 테스트

## 테스트 1: 최초 실행

business-day-context가 없는 상태.

확인:

- 오늘 로컬 날짜가 선택됨
- /closing에 오늘 날짜 표시
- 오류 없음

## 테스트 2: 날짜 변경

/closing에서 날짜를 다른 날짜로 변경한다.

확인:

- business-day-context 저장
- 화면 즉시 변경

## 테스트 3: 새로고침

날짜 변경 후 새로고침.

확인:

- 선택 날짜 유지

## 테스트 4: 화면 간 일치

다음 화면 모두 같은 영업일을 표시해야 한다.

/closing
/closing/sales
/closing/sales/baemin
/closing/sales/coupang-eats
/closing/sales/yogiyo
/closing/sales/ddangyo
/closing/sales/general

## 테스트 5: 잘못된 JSON

다음 값을 저장한다.

business-day-context = "{broken"

확인:

- 오류 없음
- 오늘 날짜로 복구

## 테스트 6: 잘못된 날짜

예:

{
  "version": 1,
  "selectedBusinessDate": "2026-02-30"
}

확인:

- 오늘 날짜로 복구

## 테스트 7: UTC 날짜 밀림

한국 시간 기준 자정 전후에도 로컬 날짜가 선택되는지 확인한다.

toISOString().slice(0, 10)만으로 오늘 날짜를 만들지 않았는지 확인한다.

## 테스트 8: 기존 매출 기능

기존 매출 입력·저장·정산 결과가 이전과 동일하게 작동하는지 확인한다.

- 매출 저장
- 예상 공제액
- 예상 정산금액
- 매출 확인 완료

## 테스트 9: 모바일

확인:

- date 입력 터치 가능
- 가로 스크롤 없음
- 날짜 텍스트 겹침 없음
- 카드 클릭 영역 유지
