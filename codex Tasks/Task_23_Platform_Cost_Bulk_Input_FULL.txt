# 7. 기존 저장값 불러오기

마운트 시 선택 날짜의 platform-cost 거래를 조회한다.

getExpensesByBusinessDateAndGroup(
  businessDate,
  "platform-cost"
)

각 거래의:

- itemId
- platformId
- amount
- transactionType

를 기준으로 입력 state를 복원한다.

이번 화면에서 직접 수정 가능한 것은:

transactionType === "expense"

인 거래만이다.

refund·cancellation 거래가 존재하면 합계에는 반영하되 입력 필드 덮어쓰기 대상으로 사용하지 않는다.

완료 보고에서 해당 정책을 명시한다.

# 8. 중복 거래 처리 정책

같은 날짜에 같은 itemId + platformId + transactionType expense 조합이 여러 건 존재할 수 있다.

이번 일괄 입력 화면에서는 최종적으로 각 조합당 expense 거래를 최대 1건으로 정규화한다.

저장 시:

```text
선택 날짜의 기존 platform-cost expense 거래
→ 화면 입력값 기준으로 교체
```

단, 기존 refund·cancellation 거래는 보존한다.

즉 교체 대상:

group === "platform-cost"
&& transactionType === "expense"

보존 대상:

group !== "platform-cost"
또는
transactionType === "refund"
또는
transactionType === "cancellation"

기존 `replaceExpenseTransactionsByGroup()`이 그룹 전체를 교체하므로 refund·cancellation까지 삭제할 위험이 있다면 그대로 사용하지 않는다.

다음 중 안전한 방식을 구현한다.

권장:

- 기존 platform-cost 거래 조회
- refund·cancellation 거래 보존
- 새 expense 거래 생성
- 보존 거래 + 새 expense 거래를 `replaceExpenseTransactionsByGroup()`에 전달

이때 같은 날짜·그룹 조건을 지킨다.

# 9. 저장 거래 생성

각 항목·플랫폼 조합에서 금액이 0원보다 큰 경우에만 거래를 생성한다.

거래 구조:

{
  id,
  businessDate,
  group: "platform-cost",
  itemId,
  itemName,
  amount,
  transactionType: "expense",
  platformId,
  createdAt,
  updatedAt
}

속성:

- vendorName 사용하지 않음
- memo 기본 사용하지 않음
- estimatedInputVat는 이번 Task에서 생성하지 않음
- inventoryApplied 사용하지 않음
- evidence/OCR 사용하지 않음

itemName은 현재 카탈로그 label을 거래 저장 당시 복사한다.

# 10. 기존 ID와 createdAt 보존

같은 날짜·itemId·platformId의 기존 expense 거래가 있고 금액만 수정된 경우:

- 기존 id 유지
- 기존 createdAt 유지
- updatedAt 갱신

신규 조합:

- 새 ID 생성
- createdAt과 updatedAt 현재 시각

금액을 0원으로 변경하면 해당 조합의 기존 expense 거래는 제거한다.

따라서 UI에서 별도의 삭제 버튼 없이 0원 저장으로 삭제 효과를 제공한다.

사용자에게 다음 보조 안내를 표시한다.

0원으로 저장하면 해당 플랫폼 비용 기록이 삭제됩니다.

# 11. 통합 저장

페이지 하단 버튼:

플랫폼 비용 저장

클릭 시:

1. 모든 입력 검증
2. 기존 platform-cost 거래 조회
3. 기존 refund·cancellation 거래 보존
4. 기존 expense 거래를 조합별 map으로 구성
5. 0원 초과 입력만 거래 생성
6. 기존 조합은 id·createdAt 보존
7. 신규 조합은 새 ID 생성
8. replaceExpenseTransactionsByGroup() 호출
9. 저장 성공 후 setExpenseUnconfirmed(businessDate)
10. 목록·합계·state 재조회
11. 성공 메시지 표시

성공 문구:

플랫폼 비용을 저장했습니다.

실패 문구:

플랫폼 비용을 저장하지 못했습니다. 다시 시도해주세요.

저장 성공 후 자동으로 비용 메인 화면으로 이동하지 않는다.

사용자가 결과를 확인할 수 있도록 현재 페이지에 머문다.

별도 `비용으로 돌아가기` 링크는 상단에 유지한다.

# 12. 비용 상태 해제

플랫폼 비용 저장이 실제로 성공하면:

setExpenseUnconfirmed(businessDate)

호출.

결과:

- expenseStatus → unconfirmed
- closingStatus → open
- completedAt → null

입력값이 기존 저장값과 완전히 같아 실제 변경이 없는 경우에도 저장 버튼을 누르면 unconfirmed로 돌릴지 정책을 정한다.

권장:

- 저장 전 기존 정규화 값과 신규 정규화 값을 비교
- 실제 데이터 변경이 있을 때만 setExpenseUnconfirmed()
- 변경 없음이면 상태 유지
- 성공 문구는 `변경된 내용이 없습니다.`

복잡도가 지나치게 커지면 저장 성공 시 항상 unconfirmed 처리할 수 있으나, 완료 보고에 명확히 적는다.

가능하면 실제 변경 감지를 구현한다.

이번 MVP에서는 최소한 과거 거래 합계가 전체 합계에서 누락되지 않게 한다.

# 16. 신규 입력·조회·수정·삭제 흐름

## 신규 입력

빈 날짜
→ 광고비와 쿠폰비 각 플랫폼 금액 입력
→ 저장
→ 플랫폼별 거래 생성

## 조회

기존 날짜 재진입
→ 기존 itemId + platformId별 금액 복원

## 수정

기존 금액 변경
→ 같은 거래 ID 유지
→ 금액·updatedAt 변경

## 삭제

기존 금액을 0원으로 변경
→ 해당 expense 거래 제거

별도의 수정 모드와 삭제 버튼은 만들지 않는다.

# 17. 환급·취소 표시

기존 refund·cancellation 거래가 있으면 입력 필드와 별도로 작은 읽기 전용 영역에 표시할 수 있다.

다만 이번 Task에서 환급·취소 UI를 새로 만들지 않아도 된다.

최소 요구사항:

- 저장 시 기존 refund·cancellation 거래 보존
- 전체 합계에는 기존 calculateExpenseSummary 기준으로 반영
- expense 입력 필드에는 합산하지 않음

- 통합 저장 버튼 최소 높이 56px
- 하단 여백 충분
- 숫자 키패드 사용 가능
- 저장 중 중복 클릭 방지

# 20. 필수 브라우저 테스트

## 테스트 1: 빈 날짜

확인:

- 광고비·쿠폰비 표시
- 4개 플랫폼 각 0원
- 전체 합계 0원
- 저장 가능
- 0원 거래는 생성되지 않음

## 테스트 2: 광고비 저장

입력:

배민 30,000
쿠팡 10,000
요기요 5,000
땡겨요 0

확인:

- 거래 3건 생성
- 플랫폼 비용 합계 45,000원
- 땡겨요 거래 없음

## 테스트 3: 쿠폰비 추가

입력:

배민 20,000
쿠팡 5,000
요기요 0
땡겨요 5,000

확인:

- 전체 expense 거래 6건
- 광고비 45,000원
- 쿠폰비 30,000원
- 전체 75,000원

## 테스트 4: 재진입 조회

새로고침 또는 페이지 재진입.

확인:

- 기존 값 복원
- 합계 동일

## 테스트 5: 수정

배민 광고비:

30,000 → 40,000

확인:

- 기존 ID 유지
- createdAt 유지
- updatedAt 변경
- 전체 합계 85,000원

## 테스트 6: 0원 삭제

쿠팡 광고비:

10,000 → 0

확인:

- 해당 거래 제거
- 다른 거래 유지
- 합계 갱신

## 테스트 7: 날짜 분리

7월 16일 저장 후 7월 17일 선택.

확인:

- 7월 17일 모두 0원
- 7월 16일 재선택 시 기존 값 복원

## 테스트 8: 비용 상태 해제

expenseStatus confirmed-with-data 상태에서 실제 금액 변경 저장.

확인:

- expenseStatus unconfirmed
- closingStatus open
- completedAt null

## 테스트 9: 변경 없음 저장

기존 값 그대로 저장.

확인:

- 데이터 중복 생성 없음
- 가능하면 상태 유지
- `변경된 내용이 없습니다.` 표시

## 테스트 10: refund 보존

기존 refund 거래가 있는 상태에서 expense 일괄 저장.

확인:

- refund 거래 유지
- expense 거래만 교체
- 합계에서 refund 음수 반영

## 테스트 11: 사용자 정의 항목

enabled platform-cost 사용자 항목 존재.

확인:

- 기본 항목 뒤 표시
- 4개 플랫폼 입력 가능
- 저장·복원 가능

## 테스트 12: 비용 메인 화면 연결

플랫폼 비용 카드 클릭.

확인:

- 실제 페이지 이동
- 준비 중 문구 제거
- 저장 후 복귀 시 건수·합계 반영

## 테스트 13: 입력 검증

차단:

- 음수
- NaN
- Infinity
- 숫자 아닌 문자열

빈 문자열은 0원 처리.

## 테스트 14: 모바일

- 4개 플랫폼 행 입력 가능
- 숫자 키패드
- 긴 금액 겹침 없음
- 저장 버튼 가림 없음
- 가로 스크롤 없음
