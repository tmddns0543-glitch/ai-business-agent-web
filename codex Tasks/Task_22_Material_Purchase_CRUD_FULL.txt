AI Business Agent 2세대 웹서비스의 다음 구현 Task를 진행해줘.

# 3. 입력 필드

원재료 매입 한 건에 필요한 필드:

- 거래처명
- 매입금액
- 메모

## 거래처명

- 일반 text input
- 앞뒤 공백 제거
- 빈 값 저장 금지
- 최대 길이는 현재 프로젝트 관례에 맞게 적절히 제한
- 향후 자동완성을 붙일 수 있도록 별도 state로 관리

## 매입금액

- 숫자 입력
- 0원 초과만 저장 가능
- 음수 금지
- NaN·Infinity 금지
- inputMode="numeric"
- 기존 MoneyField를 재사용할 수 있다면 사용
- 천 단위 입력 UX 유지

## 메모

- 선택 입력
- textarea 또는 text input
- 앞뒤 공백 제거
- 빈 문자열이면 transaction에서 memo 생략 가능
- 원재료 매입에는 거래별 메모를 허용한다

# 4. 예상 매입세액 계산

다음 순수 함수를 신규 생성한다.

권장 파일:

frontend/src/lib/expense/calculate-estimated-input-vat.ts

함수:

export function calculateEstimatedInputVat(amount: number): number

규칙:

- amount가 유한한 양수일 때 Math.round(amount / 11)
- 0 이하, NaN, Infinity는 0
- React·LocalStorage에 의존하지 않음

예:

350000 → 31818
120000 → 10909
0 → 0

사용자 화면에서는 반드시 다음 용어를 사용한다.

예상 매입세액

보조 문구:

현재 입력 기준 단순 예상치입니다.

면세·의제매입세액공제는 이번 Task에서 적용하지 않는다.

# 5. 거래 ID 생성

신규 거래마다 고유 ID가 필요하다.

권장 신규 파일:

frontend/src/lib/expense/create-expense-transaction-id.ts

함수:

createExpenseTransactionId()

요구사항:

- 브라우저에서 안전하게 고유 ID 생성
- `crypto.randomUUID()` 사용 가능 여부 확인
- fallback 필요 시 충돌 가능성을 낮춘 문자열 생성
- 제품 코드에서 단순 배열 index를 ID로 사용하지 않음
- 새 패키지 설치 금지

ID 예:

expense-xxxxxxxx

현재 프로젝트에 공통 ID helper가 있으면 재사용한다.

# 6. 신규 거래 추가

입력 검증 통과 후 다음 형태의 `ExpenseTransaction`을 생성한다.

예:

{
  id,
  businessDate,
  group: "material-purchase",
  itemId: "material-purchase.purchase",
  itemName: "원재료 매입",
  amount,
  transactionType: "expense",
  vendorName,
  memo,
  estimatedInputVat,
  taxTreatment: "taxable",
  inventoryApplied: false,
  createdAt,
  updatedAt
}

주의:

- itemId와 itemName은 `expense-default-items.ts`의 공식 정의를 조회해 사용한다.
- 문자열을 화면에 중복 하드코딩하지 않는다.
- taxTreatment 기본값은 현재 타입과 공식 규칙을 확인해 적절히 선택한다.
- 면세 UI는 만들지 않는다.
- inventoryApplied는 false.
- evidence/OCR 필드는 이번 Task에서 사용하지 않는다.

저장:

addExpenseTransaction(transaction)

성공 시:

- 입력 폼 초기화
- 거래 목록 재조회
- 요약 재계산
- 비용 상태를 unconfirmed로 변경
- 최종 마감 상태 open
- 성공 안내 표시

성공 문구:

원재료 매입을 저장했습니다.

실패 문구:

원재료 매입을 저장하지 못했습니다. 다시 시도해주세요.

# 7. 기존 거래 목록

선택 날짜의 다음 거래만 조회한다.

getExpensesByBusinessDateAndGroup(

권장:

최근 입력이 위 또는 아래 중 하나를 선택하고 완료 보고에서 명시한다.

# 9. 삭제

삭제 버튼 클릭 시 바로 삭제하지 않는다.

브라우저 confirm 또는 현재 프로젝트 스타일에 맞는 확인 UI를 사용한다.

확인 문구:

이 원재료 매입 기록을 삭제할까요?

삭제 시:

removeExpenseTransaction(businessDate, expenseId)

성공 후:

- 목록 재조회
- 요약 재계산
- 비용 상태 unconfirmed
- 최종 마감 reopen

성공 문구:

원재료 매입 기록을 삭제했습니다.

삭제 대상이 현재 수정 중인 거래라면 수정 모드를 종료한다.

# 10. 거래 변경 시 마감 상태 처리

원재료 거래의 추가·수정·삭제가 성공하면 반드시 다음 상태를 적용한다.

setExpenseUnconfirmed(businessDate)

이 함수가 다음을 처리해야 한다.

- expenseStatus → unconfirmed
- closingStatus → open
- completedAt → null
- updatedAt 갱신

즉 기존 상태가:

confirmed-with-data
confirmed-none
completed

중 무엇이든, 거래가 바뀌면 다시 비용 확인이 필요하다.

이 규칙은 저장 실패 시 적용하면 안 된다.

# 13. 0원 거래 정책

원재료 매입은 0원 거래를 저장하지 않는다.

검증:

amount > 0

빈 거래처명이나 0원 입력에서는 저장 버튼을 비활성화하거나 오류 표시한다.

권장 오류 문구:

거래처명을 입력해주세요.
매입금액은 0원보다 커야 합니다.

# 14. 환급·취소

이번 화면에서는 `transactionType` 선택 UI를 만들지 않는다.

신규·수정 거래는 모두:

transactionType: "expense"

로 저장한다.

환급·취소 입력 UI는 향후 별도 기능으로 미룬다.

기존 저장소가 refund/cancellation 거래를 읽을 수 있다면 목록에는 정상 표시하되, 이번 폼에서 수정할 수 있는지 정책을 명확히 한다.

권장:

- material-purchase 그룹의 기존 모든 거래를 목록에 표시
- 이번 화면에서 수정은 expense 타입만 허용
- refund/cancellation은 읽기 전용 또는 향후 기능 안내

구현 복잡도가 커지면 이번 MVP에서는 저장된 expense 타입만 편집 대상으로 제한하고 완료 보고에 명시한다.

# 15. 입력 데이터 불변성과 안전성

- 저장소 반환 객체 mutate 금지

- Invalid Date 표시 금지

- 긴 거래처명에서 레이아웃 깨짐 방지

현재 비용 메인 화면과 `/closing`의 얇고 단정한 UI를 따른다.

- max-w-md
- rounded-xl

- 중첩 카드 최소화
- 입력 높이 최소 52px

# 17. 필수 브라우저 테스트

## 테스트 1: 빈 날짜

선택 날짜에 원재료 거래 없음.

확인:

- 합계 0원
- 예상 매입세액 0원
- 신규 입력 폼 정상

## 테스트 2: 신규 거래

입력:

거래처: 짜르푸드
매입금액: 350,000원
메모: 닭고기·소스 입고

확인:

- 거래 저장
- 예상 매입세액 31,818원
- 목록 표시
- 합계 350,000원
- expenseStatus unconfirmed
- closingStatus open

## 테스트 3: 두 번째 거래

대명축산 120,000원 저장.

확인:

- 두 거래 유지
- 총 470,000원
- 예상 매입세액 합계 42,727원

## 테스트 4: 날짜 분리

7월 16일 거래 저장 후 7월 17일 선택.

확인:

- 7월 17일 목록 비어 있음
- 7월 16일 재선택 시 기존 거래 복원

## 테스트 5: 수정

짜르푸드 금액:

350,000 → 380,000

- 같은 ID 유지
- createdAt 유지

- 예상 매입세액 34,545원
- 합계 갱신

## 테스트 6: 삭제

대명축산 거래 삭제.

확인:

- 해당 거래만 삭제
- 다른 거래 유지
- 합계 갱신
- 상태 unconfirmed
- closingStatus open

## 테스트 7: 입력 검증

다음 입력 차단:

- 빈 거래처명
- 0원
- 음수
- NaN
- Infinity
- 숫자가 아닌 문자열

## 테스트 8: 비용 상태 해제

비용 상태 confirmed-with-data에서 거래 수정.

확인:

- expenseStatus unconfirmed
- 최종 마감 completed였다면 open
- completedAt null

## 테스트 9: confirmed-none 상태에서 거래 추가

비용 없음 확인 상태에서 거래 추가.

확인:

- 거래 저장
- expenseStatus unconfirmed

## 테스트 10: 비용 메인 화면 연결

원재료 카드 클릭.

- 빈 목록

## 테스트 12: 모바일

- 거래처·금액·메모 입력 가능
- 수정·삭제 버튼 터치 가능
