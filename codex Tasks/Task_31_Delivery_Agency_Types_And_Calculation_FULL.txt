AI Business Agent 2세대 웹서비스의 다음 구현 Task를 진행해줘.

이번 Task에서는 배달대행사 기능의 첫 번째 기반 작업만 진행한다.

이번 Task의 목적은 배달대행사 거래를 일반 비용과 분리하여, 캐시 잔액과 실제 비용이 이중 계산되지 않도록 공식 데이터 구조와 계산 규칙을 정의하는 것이다.

아직 LocalStorage 저장소와 화면은 만들지 않는다.

# 핵심 원칙

배달대행사 캐시는 선불 충전 잔액이다.

따라서:

- 캐시충전은 현금 유출이지만 비용이 아니다.
- 배달대행비는 실제 비용이며 캐시 잔액을 감소시킨다.
- 충전수수료는 실제 비용이다.
- 캐시입금은 캐시 잔액을 증가시키는 보정 또는 유입이다.
- 월정액수수료는 실제 비용이다.
- 캐시충전과 배달대행비를 모두 비용으로 잡으면 이중 계산되므로 절대 그렇게 처리하면 안 된다.

예:

캐시 300,000원 충전
배달대행비 80,000원 사용

결과:

- 현금 유출: 300,000원
- 실제 비용: 80,000원
- 캐시 잔액 증가 후 사용: +300,000 - 80,000
- 비용 합계에 380,000원을 넣으면 안 됨

# 6. 공식 재무 규칙

## 6.1 캐시충전

type:

cash-charge

의미:

- 은행 또는 카드에서 배달대행사 캐시로 자금 이동
- 현금 유출
- 비용 아님
- 캐시 잔액 증가

규칙:

- cashBalanceEffect: increase
- cashFlowEffect: outflow
- affectsOperatingExpense: false
- estimatesInputVat: false

예:

100,000원 충전

- cashBalance +100,000
- cashOutflow +100,000
- operatingExpense +0

## 6.2 충전수수료

type:

charge-fee

의미:

- 캐시 충전 과정에서 발생한 수수료
- 실제 비용
- 현금 유출
- 캐시 잔액에는 직접 반영하지 않음

규칙:

- cashBalanceEffect: none
- cashFlowEffect: outflow
- affectsOperatingExpense: true
- estimatesInputVat: false

예:

100,000원 충전
충전수수료 1,000원

- 캐시 잔액 +100,000
- 현금 유출 101,000
- 비용 1,000

## 6.3 배달대행비

type:

delivery-fee

의미:

- 실제 배달 건에 사용된 대행비
- 실제 비용
- 캐시 잔액 감소
- 충전 시점의 현금 유출과는 별도로 비용 인식

규칙:

- cashBalanceEffect: decrease
- cashFlowEffect: none
- affectsOperatingExpense: true
- estimatesInputVat: true

중요:

cashFlowEffect를 outflow로 두면 캐시충전과 이중 현금 유출이 된다.

배달대행비는 이미 충전된 캐시 사용이므로 cashFlowEffect는 none으로 둔다.

예:

캐시충전 300,000
배달대행비 80,000

- 현금 유출 300,000
- 비용 80,000
- 캐시 잔액 +220,000

## 6.4 캐시입금

type:

cash-credit

의미:

- 대행사 보상, 환급, 수동 조정 또는 별도 캐시 입금
- 캐시 잔액 증가
- 비용 아님
- 외부 현금 유입으로 단정하지 않음

규칙:

- cashBalanceEffect: increase
- cashFlowEffect: none
- affectsOperatingExpense: false
- estimatesInputVat: false

주의:

캐시입금이 실제 은행 입금인지, 대행사의 보상인지 상황이 다를 수 있다.

이번 MVP에서는 캐시 잔액 보정 성격으로만 처리한다.

현금 유입으로 계산하지 않는다.

## 6.5 월정액수수료

type:

monthly-fee

의미:

- 배달대행사 이용 월정액
- 실제 비용
- 캐시에서 차감되는 구조를 우선 가정
- 캐시 잔액 감소

규칙:

- cashBalanceEffect: decrease
- cashFlowEffect: none
- affectsOperatingExpense: true
- estimatesInputVat: true

주의:

실제 월정액이 계좌에서 별도 출금되는 대행사도 있을 수 있다.

이번 MVP에서는 캐시 차감 기준으로 정의한다.

향후 paymentSource 필드 확장을 허용한다.

# 8. 매입부가세 추정 규칙

사용자가 앞서 확정한 기준:

- 배달대행비
- 월정액수수료

는 금액을 11로 나누어 예상 매입세액으로 활용한다.

계산식:

Math.round(amount / 11)

다만 이번 Task에서는 실제 계산 함수를 별도 파일로 만들지 않아도 된다.

타입과 규칙에:

estimatesInputVat: true

를 정확히 표시한다.

충전수수료는 이번 MVP에서는 예상 매입세액 대상에서 제외한다.

향후 증빙 기준으로 수정 가능해야 한다.

# 9. 배달대행사 요약 타입

다음 타입을 정의한다.

export interface DeliveryAgencySummary {
  transactionCount: number;

  cashChargeTotal: number;
  chargeFeeTotal: number;
  deliveryFeeTotal: number;
  cashCreditTotal: number;
  monthlyFeeTotal: number;

  operatingExpenseTotal: number;
  externalCashOutflowTotal: number;
  estimatedInputVatTotal: number;

  openingCashBalance: number;
  cashBalanceChange: number;
  closingCashBalance: number;
}

의미:

- operatingExpenseTotal
  - 충전수수료 + 배달대행비 + 월정액수수료

- externalCashOutflowTotal
  - 캐시충전 + 충전수수료
  - 월정액수수료는 캐시 차감 기준이므로 외부 현금 유출에 다시 포함하지 않음

- cashBalanceChange
  - 캐시충전
  + 캐시입금
  - 배달대행비
  - 월정액수수료

- closingCashBalance
  - openingCashBalance + cashBalanceChange

- estimatedInputVatTotal
  - 배달대행비 ÷ 11
  + 월정액수수료 ÷ 11

# 11. 금액 정규화

- amount가 음수면 0
- NaN·Infinity면 0
- openingCashBalance가 NaN·Infinity면 0
- openingCashBalance는 음수 허용 여부를 명확히 정한다

권장:

openingCashBalance는 음수도 허용한다.

이유:

대행사 미납 또는 후불 잔액이 존재할 수 있다.

거래 amount는 항상 0 이상의 절댓값.

# 12. 계산 순서

각 거래를 순회해 다음을 계산한다.

cash-charge:

- cashChargeTotal += amount
- externalCashOutflowTotal += amount
- cashBalanceChange += amount

charge-fee:

- chargeFeeTotal += amount
- operatingExpenseTotal += amount
- externalCashOutflowTotal += amount

delivery-fee:

- deliveryFeeTotal += amount
- operatingExpenseTotal += amount
- cashBalanceChange -= amount
- estimatedInputVatTotal += Math.round(amount / 11)

cash-credit:

- cashCreditTotal += amount
- cashBalanceChange += amount

monthly-fee:

- monthlyFeeTotal += amount
- operatingExpenseTotal += amount
- cashBalanceChange -= amount
- estimatedInputVatTotal += Math.round(amount / 11)

closingCashBalance:

openingCashBalance + cashBalanceChange

# 13. 반올림

예상 매입세액은 거래별로 계산 후 원 단위 반올림한다.

예:

배달대행비 거래 33,000원
→ 예상 매입세액 3,000원

배달대행비 거래 22,000원
→ 예상 매입세액 2,000원

합계:

5,000원

전체 합계 금액에서 한 번 나누지 않는다.

# 15. 전체 합계에서 주의할 점

배달대행사 전체 요약에서 다음은 서로 다른 개념이다.

- 실제 운영비용
- 외부 현금 유출
- 캐시 잔액

절대 하나의 `총비용`으로 합치지 않는다.

예:

캐시충전 300,000
충전수수료 1,000
배달대행비 80,000
월정액 20,000

결과:

- 외부 현금 유출: 301,000
- 실제 운영비용: 101,000
- 캐시 잔액 변화: +200,000
- 예상 매입세액: 9,091원
  - 80,000 ÷ 11 = 7,273
  - 20,000 ÷ 11 = 1,818
  - 합계 9,091

301,000 + 101,000을 총지출로 더하면 안 된다.

# 16. 향후 마감 연결 원칙

이번 Task에서는 화면 연결을 하지 않는다.

다만 향후 마감에서는 다음처럼 표시해야 한다.

배달대행사

- 현재 캐시 잔액
- 오늘 배달대행비
- 오늘 충전액
- 오늘 실제 비용

비용 화면의 operatingExpenseTotal과 배달대행사 비용을 합칠 때는:

운영비용
+
배달대행사 operatingExpenseTotal

로 연결한다.

캐시충전은 운영비용에 포함하지 않는다.

단순 현금 잔액에서는:

매출 정산금액
- 일반 비용 totalCashOutflow
- 배달대행사 externalCashOutflowTotal

처럼 현금 유출을 반영할 수 있다.

배달대행비를 다시 현금 유출로 차감하면 안 된다.
