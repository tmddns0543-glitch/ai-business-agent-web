또는 현재 TypeScript 구조에 맞는 안전한 타입을 사용한다.

# 7. 기존값 복원

마운트 시:

getSelectedBusinessDate()
→ getExpensesByBusinessDateAndGroup(
    businessDate,
    "fixed-cost"
  )

기존 거래 중 직접 입력 필드에 복원하는 대상:

transactionType === "expense"

itemId별로 금액을 복원한다.

같은 itemId의 expense 거래가 여러 건이면 화면에서는 합산 금액을 표시하고, 다음 저장 시 한 건으로 정규화한다.

refund·cancellation 거래는 입력 필드에 합산하지 않는다.

# 8. 저장 단위와 정규화

공과금·고정비는 같은 날짜·같은 itemId·transactionType expense 조합당 최대 한 건으로 저장한다.

저장 시:

기존 fixed-cost 거래 조회
→ refund·cancellation 거래 보존
→ 관리 불가능 거래 보존
→ 입력값 기준 expense 거래 재생성
→ replaceExpenseTransactionsByGroup() 호출

보존 대상:

- refund
- cancellation
- 알 수 없는 itemId의 기존 거래
- 현재 화면에서 관리할 수 없는 기존 거래

교체 대상:

- 현재 입력 카탈로그에 포함된 itemId
- transactionType === "expense"

# 9. 거래 생성 구조

0원 초과 항목만 거래를 생성한다.

구조:

{
  id,
  businessDate,
  group: "fixed-cost",
  itemId,
  itemName,
  amount,
  transactionType: "expense",
  createdAt,
  updatedAt
}

사용하지 않는 필드:

- platformId
- vendorName
- memo
- estimatedInputVat
- inventoryApplied
- evidence
- OCR

itemName은 현재 카탈로그 label을 저장 당시 복사한다.

# 10. 기존 ID와 createdAt 보존

같은 날짜·itemId의 기존 expense 거래가 있고 금액만 변경되면:

- 기존 id 유지
- 기존 createdAt 유지
- updatedAt 갱신

신규 항목:

- 새 id 생성
- createdAt과 updatedAt 현재 시각

금액을 0원으로 바꾸면 해당 itemId의 기존 expense 거래를 제거한다.

별도 삭제 버튼은 만들지 않는다.

# 11. 통합 저장

페이지 하단 버튼:

공과금·고정비 저장

클릭 흐름:

1. 모든 입력값 정규화
2. 기존 fixed-cost 거래 조회
3. 기존 refund·cancellation 거래 보존
4. 기존 expense 거래 map 생성
5. 0원 초과 항목 거래 생성
6. 기존 itemId는 id·createdAt 보존
7. 신규 itemId는 새 ID 생성
8. replaceExpenseTransactionsByGroup() 저장
9. 실제 변경 시 setExpenseUnconfirmed(businessDate)
10. 거래·state·합계 재조회
11. 성공 메시지 표시

성공:

공과금·고정비를 저장했습니다.

실패:

공과금·고정비를 저장하지 못했습니다. 다시 시도해주세요.

변경 없음:

변경된 내용이 없습니다.

저장 성공 후 자동 이동하지 않는다.

# 12. 실제 변경 감지

기존 정규화된 expense 데이터와 새 입력 데이터를 비교한다.

비교 기준:

- itemId
- amount

순서 차이는 무시한다.

실제 변경이 없으면:

- 저장소 재작성 금지
- 비용 상태 변경 금지
- 마감 상태 reopen 금지
- `변경된 내용이 없습니다.` 표시

가능하면 플랫폼 비용 화면과 동일한 비교 helper 패턴을 재사용한다.

# 13. 비용 상태 해제

실제 데이터 변경 후 저장 성공 시:

setExpenseUnconfirmed(businessDate)

결과:

- expenseStatus → unconfirmed
- closingStatus → open
- completedAt → null

confirmed-with-data, confirmed-none, completed 상태 모두 거래 변경 후 다시 확인 필요 상태가 되어야 한다.

# 14. 합계 표시

입력 중 실시간 합계를 표시한다.

공과금·고정비 합계
= 모든 입력 항목의 정규화 금액 합계

refund·cancellation은 입력 합계에는 포함하지 않는다.

저장된 전체 fixed-cost 합계는 기존 calculateExpenseSummary() 기준으로 refund·cancellation까지 반영할 수 있다.

화면에서 다음을 구분할 수 있다.

- 입력 합계
- 저장된 조정 포함 합계

조정 거래가 없다면 하나의 합계만 표시해도 된다.

환급·취소가 존재하면 작은 안내를 표시할 수 있다.

예:

환급·취소 조정 내역이 합계에 반영되어 있습니다.

환급·취소 입력 UI는 만들지 않는다.
