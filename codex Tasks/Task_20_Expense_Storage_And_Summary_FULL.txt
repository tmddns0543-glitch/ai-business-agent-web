AI Business Agent 2세대 웹서비스의 다음 구현 Task를 진행해줘.

이번 Task에서는 비용 기능의 두 번째 기반 작업을 진행한다.

구현 범위:

1. 날짜별 비용 거래 저장소
2. 사용자 정의 비용 항목 저장소
3. 날짜별 비용 메모 저장소
4. 비용 합계 계산 순수 함수
5. 잘못된 JSON·서버 환경·비정상 값 안전 처리

아직 `/closing/expenses` 화면과 카테고리별 입력 UI는 만들지 않는다.

이번 Task의 목적은 다음 단계에서 비용 메인 화면과 각 입력 화면을 안정적으로 연결할 수 있도록 저장 계층과 계산 계층을 먼저 완성하는 것이다.

## 구현 함수

### getBusinessDayExpenseStorage()

### getExpensesByBusinessDate(businessDate)

### getExpensesByBusinessDateAndGroup(businessDate, group)

- 특정 날짜·특정 그룹 거래 반환
- 공식 저장 순서 유지
- 새 배열 반환

### getExpenseById(businessDate, expenseId)

- 해당 날짜에서 ID로 거래 조회

### addExpenseTransaction(transaction)

- 신규 거래 추가

- 다른 날짜 보존
- ID 중복이면 저장 실패 또는 명확한 오류 결과
- 저장 성공 여부 반환

### updateExpenseTransaction(businessDate, expenseId, update)

- 해당 거래만 부분 수정
- id와 businessDate는 변경하지 못하게 보호
- createdAt 유지
- updatedAt 갱신
- 다른 거래·날짜 보존
- 저장 성공 여부 반환

### removeExpenseTransaction(businessDate, expenseId)

- 해당 거래만 삭제
- 다른 거래·날짜 보존
- 저장 성공 여부 반환

### replaceExpenseTransactionsByGroup(businessDate, group, transactions)

플랫폼 비용·공과금·일반 운영비처럼 한 화면에서 여러 값을 한 번에 저장할 때 사용한다.

역할:

- 해당 날짜의 특정 group 거래만 교체
- 다른 group과 다른 날짜 거래는 보존
- 전달 거래는 모두 같은 businessDate와 group이어야 함
- 0원 거래는 호출 전에 제외하거나 저장소에서 제외하는 규칙 중 하나를 명확히 선택
- 입력 객체 불변
- 저장 성공 여부 반환

이 함수가 과도하다면 현재 구조에 맞는 더 안전한 batch 함수 이름을 선택할 수 있다.

# 3. 비용 거래 검증

저장소는 최소한 다음 필드를 검증한다.

- id: 비어 있지 않은 문자열
- businessDate: 유효한 BusinessDate
- group: 공식 ExpenseGroup
- itemId: 비어 있지 않은 문자열
- itemName: 비어 있지 않은 문자열
- amount: 유한한 0 이상의 숫자
- transactionType: expense | refund | cancellation
- createdAt: 문자열
- updatedAt: 문자열

선택 필드 검증:

- vendorName: 문자열
- platformId: 공식 ExpensePlatformId
- memo: 문자열
- estimatedInputVat: 유한한 0 이상의 숫자
- evidenceId: 문자열
- evidenceType: 공식 값
- ocrStatus: 공식 값
- inventoryApplied: boolean
- taxTreatment: 공식 값

비정상 거래는 전체 저장소를 깨뜨리지 않도록 해당 거래만 무시하는 방식을 권장한다.

완료 보고에서 선택한 정책을 설명한다.

# 4. transactionType 처리 원칙

이번 Task에서 거래 합계 계산 시 다음 부호 규칙을 정의한다.

- expense: +amount
- refund: -amount
- cancellation: -amount

취소가 원거래 전체 취소가 아닌 별도 음수 거래로 저장되는 초기 구조를 전제로 한다.

합계 결과가 음수가 될 수 있으며 임의로 0으로 자르지 않는다.

저장되는 amount 자체는 항상 0 이상의 절댓값으로 유지한다.

부호는 transactionType에서 결정한다.

### taxPaymentTotal

포함:

### totalCashOutflow

각 항목의 `affectsCashFlow`를 기본 항목 정의에서 조회해 합산하는 방식이 가장 안전하다.

거래가 없으면 0.

## transactionCount
