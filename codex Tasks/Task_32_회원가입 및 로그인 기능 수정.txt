테스트 계정 생성
→ 사업장 생성
→ 2026년 6월 LocalStorage 매출 이전
→ Supabase 조회 검증
→ Git 커밋·푸시

비용·재고·Closing DB 이전은 이번 작업에 섞지 않는다.

아래 프롬프트를 Codex에 그대로 넣어.

이번 작업의 목표는 테스트 계정으로 로그인한 뒤 사업장을 만들고,
현재 브라우저 LocalStorage에 저장된 2026년 6월 매출 데이터를
Supabase sales 테이블로 안전하게 이전한 다음,
검증을 마치고 Git에 커밋·푸시할 수 있는 상태까지 완성하는 것이다.

이번 작업 범위:

1. 실제 Supabase 환경설정 확인
2. 테스트 회원가입 및 로그인
3. 최초 사업장 생성
4. 현재 사업장 결정
5. 2026년 6월 LocalStorage 매출 미리보기
6. Supabase로 Import
7. 중복 방지 및 결과 검증
8. Git 보안 점검
9. Git 커밋 및 원격 저장소 Push

중요:

- 비용, 재고, 배달대행 데이터는 이전하지 말 것
- Closing 전체를 Supabase로 이전하지 말 것
- 2026년 6월 매출만 대상으로 할 것
- LocalStorage 원본은 삭제하거나 변경하지 말 것
- `.env.local`을 Git에 포함하지 말 것
- 실제 Supabase key를 출력하거나 커밋하지 말 것
- 테스트 계정 비밀번호를 코드·문서·Git에 넣지 말 것
- `NEXT_PUBLIC_SALES_STORAGE_MODE`는 Import 검증이 끝날 때까지 `local` 유지
- Import 성공 후에도 자동으로 `supabase`로 변경하지 말 것
- 기존 정산 계산 로직과 수수료 계산 로직을 변경하지 말 것
- 원격 저장소에 Push하기 전 반드시 변경 내용을 점검할 것

==================================================
1. 현재 프로젝트 상태 진단
==================================================

다음 항목을 먼저 확인한다.

- `frontend/.env.local` 존재 여부
- Supabase URL 설정 여부
- Publishable Key 설정 여부
- `NEXT_PUBLIC_SALES_STORAGE_MODE`
- Auth client 정상 초기화 여부
- signup/login route
- onboarding 또는 사업장 생성 route
- businesses 테이블
- business_memberships 테이블
- profiles 테이블
- sales 테이블
- 관련 migration 파일
- Sales LocalStorage key와 데이터 구조
- Sales Import 관련 기존 코드
- Git 현재 branch
- Git remote
- Git status

실제 환경변수 값과 비밀번호는 출력하지 않는다.

환경설정이 placeholder이거나 누락된 상태라면
코드 작업을 계속하기 전에 사용자에게 실제 값 입력이 필요하다고 보고한다.

==================================================
2. Migration 적용 여부 확인
==================================================

다음 테이블과 정책이 실제 Supabase 프로젝트에 적용되어야 한다.

- profiles
- businesses
- business_memberships
- sales

기존 migration 파일을 읽어 다음을 확인한다.

- 적용 순서
- 외래키
- RLS 활성화
- 회원가입 후 profile 생성 방식
- 사업장 생성 권한
- membership 생성 권한
- sales CRUD 권한
- unique constraint
- created_by / updated_by 처리

Supabase CLI 연결이 되어 있고,
안전하게 대상 프로젝트를 확인할 수 있을 때만 migration 실행을 시도한다.

대상 프로젝트가 명확하지 않거나 CLI가 연결되지 않았다면
임의 프로젝트에 migration을 실행하지 말고,
사용자가 Supabase SQL Editor에서 실행해야 할 migration 파일 순서만 보고한다.

절대로 기존 운영 데이터나 테이블을 삭제하지 않는다.

==================================================
3. 테스트 회원가입 준비
==================================================

회원가입·로그인 코드를 읽고 실제 흐름을 확인한다.

목표 흐름:

회원가입
→ 이메일 인증 또는 즉시 세션 생성
→ 로그인
→ 사업장 없음 감지
→ 사업장 생성 화면
→ business 생성
→ owner membership 생성
→ 현재 사업장 확정

테스트 계정의 이메일과 비밀번호는 사용자에게 직접 입력받거나
브라우저 UI에서 사용자가 입력하도록 한다.

Codex가 임의의 실제 이메일이나 비밀번호를 코드에 넣지 않는다.

회원가입 후 이메일 인증이 필요한 경우:
- 인증 대기 화면을 명확히 표시
- 인증 완료 후 로그인하도록 안내

==================================================
4. 최초 사업장 생성 검증
==================================================

테스트 계정 로그인 후 사업장이 없으면
사업장 생성 화면으로 이동해야 한다.

최소 입력값:

- 사업장명
- 업종 또는 business type
- 사업자번호는 선택값

생성 과정에서 다음이 보장되어야 한다.

- businesses 레코드 1개 생성
- business_memberships에 owner membership 생성
- membership.user_id는 현재 auth user
- membership.business_id는 생성된 business
- role은 owner
- 현재 사업장 helper가 해당 business를 반환
- 사업장 생성 작업이 중복 실행되지 않음

가능하면 business와 membership 생성은
하나의 안전한 서버 작업 또는 DB 함수로 처리한다.

이미 같은 사용자의 사업장이 존재한다면
새 사업장을 자동으로 또 생성하지 않는다.

==================================================
5. 2026년 6월 LocalStorage 데이터 조사
==================================================

현재 브라우저 LocalStorage의 Sales 저장 구조를 읽는다.

예상 key:

`business-day-sales`

실제 구현을 기준으로 정확한 key와 구조를 확인한다.

2026년 6월 범위:

2026-06-01
~
2026-06-30

다음 값을 계산하는 Import 미리보기를 제공한다.

- 대상 날짜 수
- 대상 매출 레코드 수
- 총매출
- 플랫폼별 레코드 수
- 플랫폼별 금액
- 0원 레코드 수
- 변환 불가능 레코드 수
- 중복 예상 레코드 수
- 실제 신규 삽입 예상 수

플랫폼 및 채널 매핑은 기존 Sales mapper를 재사용한다.
별도의 중복 mapper를 새로 만들지 않는다.

LocalStorage 원본은 수정하거나 삭제하지 않는다.

==================================================
6. Import 화면 또는 실행 진입점
==================================================

기존 Import UI가 있다면 보완하고,
없다면 개발용으로 명확한 Import 화면을 추가한다.

권장 위치:

`/more/data-import`

또는 기존 관리 화면 내부의 데이터 이전 메뉴

Import 화면은 다음 순서로 구성한다.

1. 현재 로그인 계정 확인
2. 현재 사업장 확인
3. LocalStorage 2026년 6월 데이터 감지
4. Import 미리보기
5. 사용자의 명시적인 Import 버튼
6. 실행 결과
7. Supabase 검증 결과

Import 버튼은 다음 조건에서만 활성화한다.

- 로그인 완료
- 현재 사업장 존재
- Supabase 정상 설정
- 2026년 6월 데이터 존재
- 변환 가능한 데이터 존재
- Import 실행 중이 아님

페이지를 열었다는 이유만으로 자동 Import하면 안 된다.

==================================================
7. Import 규칙
==================================================

각 매출 레코드는 현재 사업장 ID와 연결한다.

필수 기준:

- business_id
- business_date
- platform
- channel
- amount

기존 sales 스키마의 다른 필드가 있다면 mapper 규칙을 따른다.

중복 기준은 현재 DB constraint와 일치시킨다.

예상 기준:

business_id
+
business_date
+
platform
+
channel

Import는 재실행해도 중복 레코드가 늘지 않아야 한다.

권장 전략:

- upsert
또는
- 기존 레코드 확인 후 insert/update

하지만 기존 사용자가 Supabase에서 수정한 데이터를
무조건 LocalStorage 값으로 덮어쓰지 않도록 주의한다.

Import 정책을 명확히 정한다.

권장 MVP 정책:

- 신규 레코드만 삽입
- 동일 unique key가 이미 존재하면 skip
- 기존 DB 레코드는 덮어쓰지 않음
- skipped 개수를 결과에 표시

Import 결과:

- scanned
- valid
- inserted
- skipped
- failed
- total amount inserted
- errors

부분 실패가 발생하면 실패 레코드와 이유를 보여주되
실제 인증정보나 민감정보는 출력하지 않는다.

==================================================
8. Import 완료 후 검증
==================================================

Import 후 Supabase에서 같은 사업장의 2026년 6월 매출을 다시 조회한다.

다음을 비교한다.

- LocalStorage 변환 가능 레코드 수
- DB 삽입 수
- DB skip 수
- DB 최종 레코드 수
- LocalStorage 총매출
- DB 총매출
- 플랫폼별 합계
- 날짜별 합계

중복 레코드도 확인한다.

검증 성공 기준:

- 신규 대상이 모두 insert 또는 skip 처리됨
- 예상하지 못한 failed 0건
- LocalStorage와 DB 총액 일치
- 플랫폼별 금액 일치
- 중복 unique key 없음
- 다른 business_id로 저장된 레코드 없음

Import를 다시 한 번 실행하여
inserted가 0이고 기존 레코드가 skip되는지 확인할 수 있도록 한다.

단, 사용자의 명시적 실행 없이 자동 재실행하지 않는다.

9. Supabase 모드 전환 전 확인
==================================================

이번 작업에서는 다음 값을 유지한다.

NEXT_PUBLIC_SALES_STORAGE_MODE=local

Import가 성공해도 자동 변경하지 않는다.

대신 향후 전환 체크리스트만 보고한다.

전환 조건:

- 회원가입 성공
- 로그인 성공
- 사업장 생성 성공
- 6월 Import 성공
- DB 총액 검증 성공
- 다른 PC 로그인 조회 성공
- Sales CRUD 검증 성공

다른 PC 검증 전에는 저장 모드를 supabase로 바꾸지 않는다.

==================================================

예를 들어 기존 데이터가:

http://localhost:3000

에서 생성됐다면 Import도 같은 주소에서 실행해야 한다.

포트가 바뀌면 LocalStorage가 보이지 않을 수 있다.

==================================================
14. Git Push
==================================================

현재 branch와 remote를 확인한다.

이미 정상적인 원격 저장소와 branch가 설정되어 있으면
검증 완료 후 push한다.

예:

git push origin 현재브랜치

다음 상황에서는 임의로 push하지 말고 보고한다.

- remote 없음
- 원격 저장소가 불명확함
- 인증 실패
- branch 보호
- merge conflict
- 예상하지 못한 대규모 변경
- secret 노출 가능성
- 테스트 또는 build 실패
- 사용자의 개인 파일이 stage됨

강제 push는 절대 사용하지 않는다.

git push --force

금지.

정확히 사용자 작업으로 분리해서 보고한다.

실제 진행 순서

Codex가 코드를 준비한 뒤에는 네가 직접 다음만 하면 돼.

1. `.env.local`에 실제 Supabase 값 입력
2. 테스트 이메일로 회원가입
3. 사업장 생성
4. 기존 6월 데이터가 입력된 브라우저에서 Import 실행
5. 금액이 맞는지 확인
6. Codex에 Git 커밋과 Push를 마무리시킴

중요한 점 하나: Git 업로드는 6월 매출 데이터 자체를 올리는 게 아니다. Git에는 데이터 이전 기능과 코드만 올리고, 실제 매출 데이터는 Supabase DB에만 저장해야 한다. `.env.local`도 절대 Git에 올라가면 안 된다.
