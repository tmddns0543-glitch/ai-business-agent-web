AI Business Agent 2세대 웹서비스의 다음 구현 Task를 진행해줘.

이번 Task에서는 비용 카테고리 중 마지막인 `세금 납부` 입력 화면을 구현한다.

이번 Task의 핵심 UX는 다음과 같다.

```text
세금 납부
→ 부가가치세·종합소득세·원천세·지방소득세·기타세금 입력
→ 사용자 정의 항목도 함께 표시
→ 한 화면에서 일괄 저장
→ 0원 항목은 저장하지 않음
→ 기존 값은 같은 항목 기준으로 수정
→ 0원 저장은 해당 항목 삭제 효과
```

중요:

이번 화면은 `예상 세금`을 입력하거나 계산하는 화면이 아니다.

여기에는 실제로 납부한 세금만 기록한다.

세금 납부는 현금흐름에는 포함되지만 일반 영업비용에는 포함하지 않는다.

구현 범위:

1. `/closing/expenses/tax-payment` 페이지 생성
2. 공식 세금 납부 5개 항목 표시
3. 활성 사용자 정의 tax-payment 항목 표시
4. 선택 영업일 기준 기존값 조회
5. 한 화면에서 일괄 입력
6. 일괄 저장
7. 기존 ID·createdAt 보존
8. 0원 저장 시 해당 항목 거래 제거
9. refund·cancellation 보존
10. 실제 변경 시 비용 확인 상태 해제
11. 최종 마감 상태 reopen
12. 비용 메인 화면의 세금 납부 카드 링크 활성화
13. 비용 6개 카테고리 모두 실제 입력 가능 상태로 전환
14. 사용자 정의 항목 추가 UI는 아직 구현하지 않음

4. 지방소득세

# 4. 세금 납부의 회계적 의미

이 화면의 거래는 다음 의미를 가진다.

- 실제 현금 유출
- 일반 영업손익에는 포함하지 않음
- 예상 세금 계산에는 직접 포함하지 않음

기존 기본 항목의 재무 영향 규칙을 유지한다.

tax-payment:

- affectsCashFlow: true
- affectsOperatingProfit: false
- affectsTaxEstimate: false

이번 Task에서 calculateExpenseSummary()의 규칙을 변경하지 않는다.

비용 메인 화면에서도:

- 오늘 운영비용
- 실제 세금 납부
- 오늘 총 현금 지출

이 구분이 그대로 유지되어야 한다.

실제로 납부한 세금만 입력해주세요.

- 예상 부가가치세
- 예상 종합소득세
- 세율 계산
- 세금 준비금
- 세금 반영 전·후 순이익
- 신고 일정

# 6. 사용자 정의 항목

다음 카탈로그 helper를 사용한다.

getExpenseItemsForGroup("tax-payment")

표시 순서:

공식 기본 항목
→ 활성 사용자 정의 항목

비활성 사용자 항목은 신규 입력 영역에 표시하지 않는다.

단, 현재 날짜에 비활성 사용자 항목의 기존 거래가 있으면 데이터가 사라져 보이면 안 된다.

권장 처리:

- 기존 거래가 있는 비활성 사용자 항목은 `과거 항목` 영역에 표시
- itemName을 label로 사용
- 금액 수정과 0원 삭제 가능
- 저장 시 거래 보존 또는 변경 가능

현재 카탈로그에 없는 unknown itemId 거래는 최소한 읽기 전용으로 표시하고 저장 시 삭제하지 않는다.

이번 Task에서는 사용자 정의 항목 추가·수정·비활성화 UI를 만들지 않는다.

# 7. 화면 구조

예:

부가가치세             [1,250,000원]
종합소득세                    [0원]
원천세                    [85,000원]
지방소득세                 [8,500원]
기타세금                       [0원]

하단:

실제 세금 납부 합계 1,343,500원

0원으로 저장하면 해당 항목 기록이 삭제됩니다.

[세금 납부 저장]

개별 메모 입력은 만들지 않는다.

세금 납부 관련 특이사항은 기존 비용 메인 화면의 날짜별 비용 메모를 사용한다.

# 8. 입력 state

항목별 문자열 state를 사용한다.

개념:

Record<ExpenseItemId, string>

또는 현재 구조에 맞는 안전한 타입.

`any` 사용 금지.

입력 규칙:

- inputMode="numeric"
- 천 단위 쉼표
- 빈 문자열 → 0원
- 0원 이상
- 음수 금지
- 소수 입력 금지
- NaN·Infinity 금지
- 숫자가 아닌 문자는 제거 또는 저장 차단
- 매우 큰 금액에서 UI 깨짐 방지

가능하면 labor-cost 화면의 금액 입력 패턴을 재사용한다.

# 9. 기존값 복원

마운트 시:

getSelectedBusinessDate()
→ getExpensesByBusinessDateAndGroup(
    businessDate,
    "tax-payment"
  )

입력 필드에 복원하는 대상:

transactionType === "expense"

itemId별로 금액을 복원한다.

같은 itemId의 expense 거래가 여러 건이면 화면에는 합산 금액을 표시하고 다음 저장 시 한 건으로 정규화한다.

refund·cancellation 거래는 입력 필드에 합산하지 않는다.

# 10. 저장 단위와 정규화

같은 날짜·같은 itemId·transactionType expense 조합당 최대 한 건으로 저장한다.

저장 흐름:

기존 tax-payment 거래 조회
→ refund·cancellation 거래 보존
→ unknown·관리 불가능 거래 보존
→ 입력값 기준 expense 거래 재생성
→ replaceExpenseTransactionsByGroup() 호출

교체 대상:

- 현재 입력 가능한 카탈로그 항목
- transactionType === "expense"

보존 대상:

- refund
- cancellation
- unknown itemId
- 현재 화면에서 관리할 수 없는 기존 거래

# 11. 거래 생성 구조

0원 초과 항목만 거래를 생성한다.

구조:

{
  id,
  businessDate,
  group: "tax-payment",
  itemId,
  itemName,
  amount,
  transactionType: "expense",
  createdAt,
  updatedAt
}

사용하지 않는 필드:

- platformId
- vendorName
- memo
- estimatedInputVat
- inventoryApplied
- evidence
- OCR

itemName은 현재 카탈로그 label을 저장 당시 복사한다.

# 12. 기존 ID와 createdAt 보존

같은 날짜·itemId의 기존 expense 거래가 있고 금액만 변경되면:

- 기존 id 유지
- 기존 createdAt 유지
- updatedAt 갱신

신규 항목:

- 새 ID 생성
- createdAt과 updatedAt 현재 시각

금액을 0원으로 바꾸면 해당 itemId의 기존 expense 거래를 제거한다.

별도 삭제 버튼은 만들지 않는다.

# 13. 통합 저장

페이지 하단 버튼:

세금 납부 저장

클릭 흐름:

1. 입력값 정규화
2. 기존 tax-payment 거래 조회
3. refund·cancellation 보존
4. unknown 거래 보존
5. 기존 expense 거래 map 구성
6. 0원 초과 항목 거래 생성
7. 기존 itemId는 id·createdAt 보존
8. 신규 itemId는 새 ID 생성
9. 실제 변경 여부 비교
10. 변경 시 replaceExpenseTransactionsByGroup() 저장
11. 저장 성공 후 setExpenseUnconfirmed(businessDate)
12. 거래·state·합계 재조회
13. 성공 메시지 표시

성공:

세금 납부 내역을 저장했습니다.

실패:

세금 납부 내역을 저장하지 못했습니다. 다시 시도해주세요.

변경 없음:

변경된 내용이 없습니다.

저장 성공 후 현재 페이지에 머문다.

# 14. 실제 변경 감지

기존 정규화된 expense 데이터와 신규 입력 데이터를 비교한다.

비교 기준:

- itemId
- amount

순서 차이는 무시한다.

변경이 없으면:

- LocalStorage 재작성 금지
- 비용 상태 변경 금지
- 최종 마감 reopen 금지
- `변경된 내용이 없습니다.` 표시

가능하면 labor-cost 페이지와 동일한 변경 감지 패턴을 사용한다.

# 15. 비용 상태 해제

실제 데이터 변경 후 저장 성공 시:

setExpenseUnconfirmed(businessDate)

결과:

- expenseStatus → unconfirmed
- closingStatus → open
- completedAt → null

confirmed-with-data, confirmed-none, completed 상태에서 값이 변경되면 다시 비용 확인이 필요하다.

# 16. 합계 표시

입력 중 실시간 합계를 표시한다.

실제 세금 납부 합계
= 모든 입력 항목 정규화 금액 합계

refund·cancellation은 입력 합계에는 포함하지 않는다.

저장된 tax-payment 전체 합계는 기존 calculateExpenseSummary() 기준으로 조정 거래를 반영한다.

refund·cancellation이 존재하면 보조 안내 가능:

환급·취소 조정 내역이 저장된 세금 납부 합계에 반영되어 있습니다.

환급·취소 입력 UI는 만들지 않는다.

# 17. 비용 메인 화면 링크 활성화

수정 파일:

frontend/src/app/closing/expenses/page.tsx

세금 납부 카드의 `준비 중`을 제거한다.

경로:

/closing/expenses/tax-payment

표시:

부가가치세·종합소득세·원천세 등
3건 · 1,343,500원
>

기존 활성 링크 유지:

- 원재료 매입
- 플랫폼 비용
- 공과금·고정비
- 일반 운영비

이번 Task 완료 후 6개 비용 카테고리 카드가 모두 실제 입력 화면으로 연결되어야 한다.

# 18. 비용 메인 화면 요약 검증

세금 납부 입력 후 `/closing/expenses`로 돌아왔을 때 다음 값이 정확해야 한다.

예:

운영비용 5,000,000원
실제 세금 납부 1,343,500원
오늘 총 현금 지출 6,343,500원

세금 납부 금액이 운영비용에 합산되면 안 된다.

세금 납부는 다음에만 포함한다.

- taxPaymentTotal
- totalCashOutflow
- byGroup["tax-payment"]

다음에는 포함하지 않는다.

- operatingExpenseTotal

# 19. 마감 화면 비용 카드 검증

`/closing` 비용 카드에서도 기존 요약 규칙을 유지한다.

권장 표시:

오늘 운영비용 5,000,000원
실제 세금 납부 1,343,500원

단순 잔액 계산에서는 totalCashOutflow를 사용하므로 실제 세금 납부도 현금 유출에 반영되어야 한다.

# 20. 세금 입력 단위 정책

이번 MVP에서는 실제 납부일 기준으로 입력한다.

예:

7월 부가가치세를 7월 25일 납부
→ 7월 25일 영업일에 입력

귀속 기간, 신고 대상 기간, 과세 기간은 입력하지 않는다.

향후 세금 분석 기능에서 별도 확장한다.

# 21. refund·cancellation 보존

기존 refund·cancellation 거래가 있는 상태에서 저장해도 삭제되면 안 된다.

최소 요구사항:

- refund·cancellation 보존
- 입력 필드에 합산하지 않음
- 저장된 합계에는 음수 반영
- 별도 입력 UI 없음

# 22. unknown 항목 보존

현재 카탈로그에 없는 과거 거래를 저장 시 삭제하지 않는다.

최소한:

- 전체 합계에서 누락되지 않음
- 저장 시 그대로 보존
- 가능하면 과거 항목 영역에서 label·금액 표시

# 23. 오류 처리

다음 경우 화면이 깨지지 않아야 한다.

- businessDate 오류
- 깨진 비용 JSON
- 깨진 사용자 항목 JSON
- itemId 누락
- 중복 itemId expense 거래
- NaN·Infinity
- 매우 큰 금액
- LocalStorage 저장 실패
- 비활성 사용자 항목 거래
- unknown itemId 거래

# 24. 모바일 UI 원칙

현재 비용 UI의 얇고 단정한 스타일을 유지한다.

- max-w-md
- rounded-xl
- 항목 행 최소 52px

# 25. 필수 브라우저 테스트

## 테스트 1: 빈 날짜

확인:

- 공식 5개 항목 표시
- 모두 0원
- 합계 0원
- 저장 시 거래 생성 없음

## 테스트 2: 기본 입력

입력:

부가가치세 1,250,000
원천세 85,000
지방소득세 8,500

확인:

- 거래 3건
- 세금 납부 합계 1,343,500원

## 테스트 3: 비용 요약 분리

동일 날짜 운영비용이 5,000,000원인 상태.

확인:

- operatingExpenseTotal 5,000,000원 유지
- taxPaymentTotal 1,343,500원
- totalCashOutflow 6,343,500원

## 테스트 4: 재진입 조회

새로고침 또는 재진입.

확인:

- 기존 값 복원
- 합계 동일

## 테스트 5: 수정

부가가치세:

1,250,000 → 1,300,000

확인:

- 기존 ID 유지
- createdAt 유지
- updatedAt 변경
- 합계 1,393,500원

## 테스트 6: 0원 삭제

지방소득세:

8,500 → 0

확인:

- 해당 거래 제거
- 다른 거래 유지
- 합계 갱신

## 테스트 7: 날짜 분리

7월 16일 저장 후 7월 17일 선택.

확인:

- 7월 17일 모두 0원
- 7월 16일 재선택 시 기존 값 복원

## 테스트 8: 비용 상태 해제

expenseStatus confirmed-with-data 상태에서 실제 변경 저장.

확인:

- expenseStatus unconfirmed
- closingStatus open
- completedAt null

## 테스트 9: 변경 없음

기존 값 그대로 저장.

확인:

- 거래 중복 없음
- 상태 유지
- 변경 없음 메시지

## 테스트 10: refund 보존

## 테스트 11: 사용자 정의 항목

enabled tax-payment 사용자 항목:

자동차세

확인:
