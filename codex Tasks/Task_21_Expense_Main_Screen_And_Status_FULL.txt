AI Business Agent 2세대 웹서비스의 다음 구현 Task를 진행해줘.

이번 Task에서는 비용 기능의 첫 번째 화면인 `/closing/expenses` 메인 화면을 구현한다.

이번 Task의 구현 범위:

1. 선택 영업일 기준 비용 메인 화면 생성
2. 비용 카테고리별 거래 건수·합계 표시
3. 오늘 운영비용·실제 세금 납부·오늘 총 현금 지출·예상 매입세액 표시
4. 날짜별 비용 메모 조회·저장
5. 비용 확인 완료 처리
6. 오늘 발생한 비용 없음 처리
7. `/closing` 비용 카드에 날짜별 비용 합계와 상태 반영
8. 비용 상태를 `unconfirmed | confirmed-with-data | confirmed-none`으로 확장

이번 Task에서는 카테고리별 실제 입력 화면은 만들지 않는다.

즉, `/closing/expenses` 메인 화면과 마감 상태 연결까지만 구현한다.

# 1. 비용 상태 타입 확장

수정 파일:

frontend/src/types/closing-status-by-business-day.ts

현재 비용 상태가 `unconfirmed | confirmed`라면 비용 섹션만 다음 세 상태를 표현할 수 있도록 확장한다.

권장 타입:

export type ExpenseClosingStatus =
  | "unconfirmed"
  | "confirmed-with-data"
  | "confirmed-none";

매출과 배달대행 상태는 기존 구조를 유지한다.

권장 방향:

export type ConfirmableSectionStatus =
  | "unconfirmed"
  | "confirmed";

export type ExpenseClosingStatus =
  | "unconfirmed"
  | "confirmed-with-data"
  | "confirmed-none";

export interface BusinessDayClosingStatus {
  salesStatus: ConfirmableSectionStatus;
  expenseStatus: ExpenseClosingStatus;
  deliveryStatus: ConfirmableSectionStatus;
  closingStatus: FinalClosingStatus;
  completedAt: string | null;
  updatedAt: string;
}

기존 저장 데이터의 `expenseStatus: "confirmed"`가 있을 수 있다면 개발용 임시값이므로 다음 중 하나를 선택한다.

- 잘못된 값으로 보고 `unconfirmed`로 복구
- 또는 `"confirmed"`를 `"confirmed-with-data"`로 안전하게 해석

현재 데이터가 테스트용이므로 가장 단순하고 안전한 방식을 선택하고 완료 보고에서 설명한다.

# 2. 날짜별 마감 상태 저장소 확장

수정 파일:

frontend/src/lib/storage/closing-status-by-business-day-storage.ts

다음 함수 또는 동일 역할의 함수를 추가한다.

setExpenseConfirmedWithData(businessDate)

# 8. 그룹별 거래 건수와 합계

각 그룹별로 다음을 계산한다.

- transactionCount
- signed amount total

`transactionType` 부호 규칙을 사용한다.

- expense: +
- refund: -
- cancellation: -

그룹별 합계 계산을 화면에서 중복 구현하지 않는다.

기존 `calculateExpenseSummary()`와 `byGroup`를 재사용한다.

그룹별 건수 helper가 없다면 작은 순수 helper를 추가할 수 있다.

가능한 파일:

frontend/src/lib/expense/calculate-expense-group-summary.ts

불필요한 파일 생성은 피한다.

버튼 문구:

setExpenseConfirmedWithData(businessDate)
→ 상태 재조회
→ /closing 이동

거래가 하나도 없으면 이 버튼은 비활성화하거나 숨긴다.

# 14. 완료 상태에서 데이터 변화 가능성

이번 Task에서는 카테고리 입력 화면이 없으므로 실제 거래 변경 후 상태 해제는 아직 발생하지 않는다.

다만 향후 거래가 추가·수정·삭제되면:

또는 closingStatus reopen 정책을 적용할 예정임을 코드 주석 또는 완료 보고에 명시한다.

이번 Task에서는 화면 없는 미래 기능을 과도하게 구현하지 않는다.

# 16. /closing 비용 카드 배지 규칙

선택 영업일의 expenseStatus 기준:

unconfirmed
→ 확인 필요

confirmed-with-data
→ 완료

confirmed-none
→ 없음 확인

비용 거래가 있는데 expenseStatus가 confirmed-none이면 비정상 상태다.

화면 안전 규칙:

transactions.length > 0 && expenseStatus === confirmed-none
→ 확인 필요로 표시

또는 저장 상태를 자동 수정하지 않고 화면만 안전하게 표시.

완료 보고에서 선택한 정책을 설명한다.

# 17. 진행률 계산 수정

현재 진행률 계산에서 expenseStatus가 단순 confirmed인지 확인하고 있을 수 있다.

다음 두 상태 모두 비용 완료로 계산한다.

- confirmed-with-data
- confirmed-none

즉:

expenseCompleted =
  expenseStatus === "confirmed-with-data"
  || expenseStatus === "confirmed-none"

매출과 배달대행 진행률 규칙은 변경하지 않는다.

최종 마감은 이미 조건 없이 가능하므로 진행률은 안내 정보로만 유지한다.

# 18. 최종 마감 안내의 미확인 항목 수

`/closing`의 미확인 항목 수 계산에서도 비용 완료는 다음 두 상태를 모두 인정한다.

- confirmed-with-data
- confirmed-none

비용이 없음 확인 상태인데 미확인으로 계산되면 안 된다.

# 19. 직접 주소 접근

`/closing/expenses` 직접 접근 시에도 선택 영업일을 기준으로 정상 조회해야 한다.

선택 영업일 저장값이 없거나 잘못된 경우 기존 business-day storage fallback을 사용한다.

# 20. 화면 UI 원칙

현재 `/closing`에서 정리한 디자인 방향을 따른다.

- max-w-md
- rounded-xl
